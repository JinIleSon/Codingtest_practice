## 문제 설명

짝수 `n`이 여러 개 주어질 때, 각 `n`에 대해 두 홀수 소수 `a`, `b`를 찾아 `n = a + b` 형태로 출력한다.  
입력은 `0`이 들어올 때 종료되며, 표현이 불가능하면 아래 문장을 출력한다.

```
Goldbach's conjecture is wrong.
```

---

## 입출력

**입력**
- 여러 줄에 하나씩 짝수 `n`이 주어진다.
- `0`이 입력되면 종료.

**출력**
- 각 짝수 `n`에 대해 `n = a + b` 형태로 출력.
- 표현이 불가능하면 `Goldbach's conjecture is wrong.` 출력.

**예시**

| 입력 | 출력 |
|------|------|
| `8` | `8 = 3 + 5` |
| `20` | `20 = 3 + 17` |
| `42` | `42 = 5 + 37` |
| `0` | *(종료)* |

---

## 풀이 접근

### ❌ 처음 접근 — 이중 반복문으로 모든 조합 비교

처음에는 소수 리스트를 만든 뒤 `a`와 `b`를 각각 순회하며 `a + b == n`인지 확인하려 했다.

```python
for a in decimal:
    for b in decimal:
        if a + b == n: ...
```

구현은 직관적이지만, 소수 개수가 많아질수록 비교 횟수가 기하급수적으로 늘어나 **시간 초과** 발생.

### ❌ 두 번째 문제 — 소수 판별을 매번 반복

`while True`로 입력을 받을 때마다 그 수까지 에라토스테네스의 체를 **다시 만들었다**.  
입력이 여러 개면 같은 계산을 계속 반복하게 되어 비효율적.

---

## 핵심 아이디어

### 1. 에라토스테네스의 체를 한 번만 만들기

입력을 모두 받은 뒤, **최댓값까지 소수 여부 배열을 딱 한 번만 생성**해 재사용한다.

```
입력 전체 저장 → 최댓값 파악 → 체 1회 생성 → 각 n에 재사용
```

### 2. `a`를 정하면 `b`는 자동으로 결정된다

`n = a + b` 이므로, `a`가 정해지면 `b = n - a`로 바로 결정된다.  
`b`를 다시 반복문으로 찾을 필요 없이, **`b`가 소수인지만 확인**하면 된다.

```python
b = n - a
if is_prime[b]: ...  # 끝
```

### 3. 리스트 탐색(`in`) 대신 배열 인덱스로 즉시 접근

| 방식 | 코드 | 시간 |
|------|------|------|
| 리스트 탐색 | `if b in decimal` | O(N) |
| 배열 인덱스 | `if is_prime[b]` | **O(1)** |

반복문 안에서 매번 membership check를 하는 경우, 리스트의 `in`은 성능 병목이 될 수 있다.

## 최종 코드

```python
import sys

input = sys.stdin.readline

# 입력받을 숫자 리스트
numbers = []

while True:
    n = int(input().strip())
    if n == 0:
        break
    numbers.append(n)

# 가장 큰 값 추출(체를 한 번만 돌기 위해)
maxValue = max(numbers)

# 소수만 True인 리스트
decimal = []

# 소수를 담을 숫자 리스트
numberList = []

for i in range(maxValue + 1):
    decimal.append(True)

decimal[0] = False
decimal[1] = False

i = 2
# i 2번 곱하면 maxValue보다 작아야 최소한으로 돌 수 있음
while i * i <= maxValue:
    j = 2
    # 합성수 모두를 False로 바꿈
    while i * j <= maxValue:
        decimal[i * j] = False
        j += 1
    i += 1

for i in range(len(decimal)):
    # 소수 리스트 안에 True로 찍혀있으면 소수니 numberList에 해당 숫자 추가
    if decimal[i]:
        numberList.append(i)

# 알맞은 쌍을 찾아서 print했다면 True로 변경할 변수
found = False

for n in numbers:
    # b는 n에서 a를 뺀 값에 대해 존재해야 하므로 b를 구한 다음 numberList안에 있는지 확인
    for a in numberList:
        b = n - a
        # in을 쓰면 전체 순회를 하기 때문에 인덱스로 접근
        # decimal 리스트는 소수만 True임
        if decimal[b]:
            print(f'{n} = {a} + {b}')
            found = True
            break
if not found:
    print("Goldbach's conjecture is wrong.")
```

---

## 복잡도 분석

| 항목 | 복잡도 |
|------|--------|
| 시간 복잡도 | 체 생성: `O(N log log N)` / 각 입력 처리: `O(N)` 이하 |
| 공간 복잡도 | `O(N)` |

> `N` = 입력값 중 최댓값

- 에라토스테네스의 체는 최댓값까지 **단 한 번만 생성**
- 이후 각 입력은 `a`를 순회하며 `is_prime[b]`로 **즉시 확인**

---

## 배운 점

**입력 처리와 전처리를 분리하기**  
입력이 여러 개일 때는 매번 계산하지 말고, 전체 범위를 파악한 뒤 한 번만 전처리하는 방식이 훨씬 효율적이다.

**관계식으로 변수 하나를 결정하기**  
`a + b = n` 같은 관계식이 있으면 두 변수를 모두 탐색하지 않아도 된다. `a`를 정하는 순간 `b`는 자동으로 결정된다.

**리스트 탐색과 배열 인덱스 접근의 차이**  
`in`은 편하지만 반복 호출 시 느릴 수 있다. 소수 여부 배열처럼 인덱스로 바로 접근하는 방식이 반복문 내에서는 훨씬 적절하다.

**시간 초과는 구조 문제일 수 있다**  
단순히 `for`를 `while`로 바꾸는 게 아니라, 입력 처리 구조와 자료 접근 방식 전체를 다시 봐야 하는 경우가 있다.

> 입력을 먼저 모은 뒤 체를 한 번만 만들고 `a`를 정하면 `b = n - a`로 바로 확인할 수 있다는  
> 구조를 이해하면서 문제를 훨씬 단순하게 볼 수 있게 되었다.
