## 문제 설명

각 테스트 케이스마다 여러 개의 수가 주어졌을 때, **가능한 모든 두 수의 쌍**에 대해 각각 GCD를 구하고 **전부 더한 값**을 출력한다.

예를 들어 `10 20 30 40`이 주어지면 쌍은 다음과 같다.

| 쌍 | GCD |
|----|-----|
| (10, 20) | 10 |
| (10, 30) | 10 |
| (10, 40) | 10 |
| (20, 30) | 10 |
| (20, 40) | 20 |
| (30, 40) | 10 |

→ 합계: **70**

---

## 입출력

**입력**
- 첫째 줄: 테스트 케이스 수 `T`
- 각 테스트 케이스: 맨 앞 숫자 `n` (쌍의 개수), 이후 `n`개의 정수

**출력**
- 각 테스트 케이스마다 모든 쌍의 GCD 합 출력

**예시**

```
입력          출력
3
4 10 20 30 40   70
3 7 5 12         0
3 125 15 25     35
```

## 헷갈렸던 부분

**1. "가능한 모든 쌍"의 의미**

처음에는 여러 수에 대해 GCD를 한꺼번에 구하는 건가 생각했다. 실제로는 두 수씩 뽑는 **조합(Combination)** 이었다.

```
(a, b) 는 봄
(a, b, c) 는 안 봄
(a, b) 와 (b, a) 는 같은 쌍 → 한 번만 봄
```

**2. 입력 형식 해석**

한 줄이 `4 10 20 30 40`처럼 들어오면, 처음에는 `4`도 계산 대상처럼 보였다.  
실제로는 맨 앞 숫자는 **뒤에 오는 숫자의 개수**이고, 나머지가 실제 계산 대상이다.

```python
data = list(map(int, input().split()))
n = data[0]        # 개수
nums = data[1:]    # 실제 숫자들
```

**3. 공통 소인수의 "개수"까지 고려해야 함**

처음에는 공통 소인수 종류만 찾으면 된다고 생각했다.  
하지만 GCD는 공통 소인수를 **공통 개수만큼** 곱해야 한다.

```
12 = 2² × 3
24 = 2³ × 3

공통 소인수: 2, 3
→ 2는 min(2, 3) = 2번, 3은 min(1, 1) = 1번
→ GCD = 2² × 3 = 12
```

set처럼 종류만 보는 것으로는 부족하고, **지수(개수)** 까지 비교해야 한다.

**4. 소인수분해 종료 조건 — `i * i <= n`**

왜 제곱근까지만 보면 되는지 처음에는 감이 안 왔다.  
핵심은 이렇다.

> 합성수라면 반드시 **제곱근 이하의 약수**가 하나 존재한다.  
> 그 구간까지 다 봤는데 안 나눠지면, 남은 값은 소수다.

소인수분해 중에는 값이 계속 줄어드므로, 원본이 아니라 **현재 남은 값 기준**으로 조건을 봐야 한다.

## 풀이 접근

### ❌ 처음 접근 — 소인수분해 후 공통 소인수 비교

각 수를 소인수분해해서 `dict`로 저장한 뒤, 쌍마다 공통 소인수를 비교해 GCD를 구하려 했다.

```python
12 → {2: 2, 3: 1}
24 → {2: 3, 3: 1}
# 공통 key 순회 → min(지수) 곱해서 GCD 계산
```

개념적으로는 맞지만 **모든 쌍을 다 비교해야 하는 이 문제**에서는 지나치게 무거웠다.

- 각 수를 소인수분해
- 모든 쌍마다 dict 순회 및 비교
- 지수 곱셈으로 GCD 재조립

이 세 단계가 반복되면서 비효율이 누적됐다.

### ✅ 최종 접근 — 모든 쌍에 GCD를 빠르게 반복 계산

풀이의 구조는 단순하다.

```
1. 테스트 케이스에서 숫자 리스트를 받는다.
2. 가능한 모든 두 수의 쌍을 만든다. (이중 반복문, i < j)
3. 각 쌍의 GCD를 빠르게 구한다.
4. GCD를 누적해서 출력한다.
```

핵심은 GCD를 **소인수분해 없이 빠르게 구하는 것**이다.  
파이썬에서는 `math.gcd`를 바로 사용할 수 있고, 유클리드 호제법으로 직접 구현할 수도 있다.

```python
# 유클리드 호제법
def gcd(a, b):
    while b:
        a, b = b, a % b
    return a
```

```python
# 파이썬 내장
from math import gcd
```

모든 쌍을 순회할 때는 `(a, b)`와 `(b, a)`를 중복으로 보지 않도록 `j > i` 조건을 사용한다.

```python
for i in range(len(nums)):
    for j in range(i + 1, len(nums)):
        total += gcd(nums[i], nums[j])
```

## 복잡도 분석

| 항목 | 처음 접근 (소인수분해) | 최종 접근 (유클리드 호제법) |
|------|----------------------|--------------------------|
| GCD 1회 계산 | `O(√N × D)` | `O(log N)` |
| 전체 쌍 처리 | 느림 | `O(n² log N)` |
| 구현 복잡도 | 높음 | 낮음 |

> `n` = 숫자 개수, `N` = 숫자 크기, `D` = dict 순회 비용

쌍을 만드는 이중 반복은 피할 수 없지만, **각 쌍의 GCD를 얼마나 빠르게 구하느냐**가 전체 성능을 좌우한다.

---

## 배운 점

**"가능한 모든 쌍"은 조합이다**  
`(a, b)`와 `(b, a)`는 같은 쌍이므로 한 번만 봐야 하고, 세 수 이상의 묶음은 포함하지 않는다.  
문제 문장의 해석을 정확히 하지 않으면 전혀 다른 방향으로 풀게 된다.

**소인수분해와 GCD는 개념은 같지만 효율은 다르다**  
공통 소인수를 공통 개수만큼 곱하면 GCD가 된다는 원리는 맞지만, 반복 계산 상황에서는 소인수분해 방식이 지나치게 비싸다.  
이 문제는 "공통 소인수를 찾는 문제"가 아니라 **"GCD를 빠르게 반복 계산하는 문제"** 였다.

**`i * i <= n` 조건의 의미를 체감했다**  
제곱근까지만 보면 된다는 것을 공식처럼 외우는 게 아니라, 합성수에는 반드시 제곱근 이하의 약수가 존재한다는 원리로 이해하게 됐다.

**정석 풀이가 왜 정석인지 납득하게 됐다**  
소인수분해 방식을 끝까지 붙잡아 보면서 한계를 직접 확인했기 때문에, 유클리드 호제법이 왜 GCD의 표준 방법인지 자연스럽게 이해됐다.

---

> GCD를 소인수분해 관점으로 끝까지 붙잡아 보면서 개념은 더 깊게 이해했지만,  
> 결국 이 문제는 "공통 소인수를 찾는 문제"가 아니라  
> **"GCD를 빠르게 반복 계산하는 문제"** 라는 걸 배웠다.
