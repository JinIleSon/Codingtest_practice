## 문제 설명

크기가 N인 수열 A가 주어졌을 때, 각 원소 A[i]에 대해 오른쪽에 있으면서 A[i]보다 큰 수 중 가장 왼쪽에 있는 수를 구하는 문제이다.

이 수를 **오큰수(NGE, Next Greater Element)** 라고 한다. 만약 오른쪽에 자신보다 큰 수가 없다면 `-1`을 출력한다.

### 입력

- 첫째 줄에 수열의 크기 N
- 둘째 줄에 수열 A₁ A₂ ... Aₙ

### 출력

- 총 N개의 수를 공백으로 구분하여 출력
- 각 위치의 오큰수를 순서대로 출력

### 입출력 예시

| 입력 | 출력 |
|------|------|
| `4`<br>`3 5 2 7` | `5 7 7 -1` |

---

## 풀이 접근

### 처음 접근 — 이중 반복문

처음에는 각 원소마다 오른쪽을 끝까지 확인하는 방식으로 접근했다.

- 현재 인덱스 `i`를 기준으로
- `j = i + 1`부터 끝까지 순회하면서
- 처음으로 더 큰 수를 만나면 출력하고
- 끝까지 못 찾으면 `-1`을 출력

이 방식은 직관적이지만, 각 원소마다 오른쪽 구간을 다시 확인하게 되므로 이중 반복문 구조가 된다.

**왜 비효율적인가**

겉보기에는 한 번 지나간 인덱스를 다시 보지 않는 것처럼 느껴질 수 있다. 하지만 실제로는 같은 인덱스가 다른 `i`의 비교 대상으로 여러 번 다시 등장한다.

```
i = 0일 때 → 오른쪽 구간 전체 확인
i = 1일 때 → 다시 그 오른쪽 구간 확인
i = 2일 때 → 또 확인
```

최악의 경우 시간 복잡도는 **O(N²)** 이 되어 시간 초과가 발생할 수 있다.

---

### 핵심 아이디어 — 스택으로 미해결 인덱스 관리

> 현재 원소의 답을 찾으러 가는 것이 아니라,  
> **현재 값이 이전의 미해결 원소들의 답이 되는지 확인**하는 방식으로 사고를 전환한다.

스택에는 **아직 오큰수가 정해지지 않은 인덱스**를 넣어둔다.

현재 인덱스 `i`를 보면서, 스택의 맨 위 인덱스가 가리키는 값이 현재 값보다 작다면 현재 값이 그 인덱스의 오큰수가 된다. 조건이 맞는 동안 계속 pop 하면서 결과를 채운다.

### 풀이 흐름

1. 수열을 왼쪽부터 오른쪽으로 한 번 순회한다.
2. 스택의 맨 위 인덱스에 해당하는 값이 현재 값보다 작다면, 현재 값이 그 인덱스의 오큰수이므로 결과 배열에 저장하고 pop 한다.
3. 조건이 맞지 않을 때까지 반복한다.
4. 현재 인덱스는 아직 자신의 오큰수를 모르는 상태이므로 스택에 넣는다.
5. 순회가 끝난 뒤에도 스택에 남아 있는 인덱스들은 끝까지 오큰수를 찾지 못한 것이므로 `-1`로 처리한다.

## 고민했던 부분

### 1. 왜 스택을 쓰는지 잘 와닿지 않았다

처음에는 스택이나 큐를 사용해도 결국 순회는 해야 하니까, 시간 복잡도가 왜 줄어드는지 감이 잘 오지 않았다.

핵심은 **순회를 하느냐가 아니라, 같은 원소를 여러 번 다시 보느냐**에 있었다.

- 이중 반복문 → 같은 인덱스를 여러 기준점에서 반복해서 비교
- 스택 방식 → 각 인덱스가 **스택에 한 번 들어가고, 최대 한 번 나오는 구조**

### 2. 큐가 아니라 왜 스택인지 헷갈렸다

`3 5 2 7` 예시를 보면 처음 들어간 `3`이 먼저 처리되는 것처럼 보여서 처음에는 큐처럼 느껴질 수 있었다.

하지만 중요한 것은 **가장 최근에 들어온 미해결 원소부터 확인**해야 한다는 점이다.

`5`를 넣고 그 뒤에 `2`를 넣었다면, `7`을 만났을 때는 먼저 `2`, 그다음 `5` 순서로 확인해야 한다. 나중에 들어온 것이 먼저 처리되는 구조이므로 **큐가 아니라 스택**이다.

### 3. 스택에 무엇을 넣어야 하는지 헷갈렸다

처음에는 값 자체를 넣어야 하는지, 인덱스를 넣어야 하는지, 특정 조건일 때만 넣어야 하는지 헷갈렸다.

정리하면 스택에 넣는 것은 큰 값도, 작은 값도, 특정 조건의 값도 아니라 **그냥 아직 오큰수가 정해지지 않은 인덱스**이다. 현재 원소는 자기 오른쪽을 아직 보지 않았기 때문에 기본적으로 미해결 상태라고 보고 스택에 들어간다.

### 4. -1을 언제 처리해야 하는지 헷갈렸다

중간에 비교가 실패하면 바로 `-1`을 넣고 싶어질 수 있다. 하지만 오큰수는 지금 당장 못 찾았다고 해서 없는 것이 아니다.

`-1`은 순회 중간에 결정하는 값이 아니라, **마지막까지 스택에 남은 인덱스들에 대해 최종적으로 처리하는 값**이다.

## 최종 코드

```python
import sys

input = sys.stdin.readline
N = int(input().strip())
number = list(map(int, input().strip().split()))

stack = []

# result에는 number와 똑같은 길이만큼의 list를 넣어줌
result = number

for i in range(N):
    # 현재 number의 i 인덱스에 대한 값이 스택 내부의 마지막 값보다 크면 pop하고 result 리스트에 추가
    while len(stack) > 0 and number[stack[len(stack)-1]] < number[i]:
        result[stack[len(stack)-1]] = number[i]
        stack.pop()
    stack.append(i)

# 스택에 인덱스가 남으면 결정이 나지 않았다는 거니까 -1로 넣기
for i in stack:
    result[i] = -1

# 출력
for i in range(len(result)):
    print(result[i], end=' ')
```

---

## 복잡도 분석

| 항목 | 복잡도 |
|------|--------|
| 시간 복잡도 | O(N) |
| 공간 복잡도 | O(N) |

겉으로 보면 `for`문 안에 `while`문이 있어서 이중 반복처럼 보일 수 있다. 하지만 각 인덱스는 스택에 한 번 들어가고 최대 한 번 pop 된다. 즉, 전체적으로 push N번, pop N번 정도만 일어나므로 총 시간 복잡도는 **O(N)** 이다.

## 이번 문제에서 얻은 것

**이중 반복문을 다른 방식으로 바꾸는 감각** — 같은 구간을 반복해서 다시 보는 구조인지, 미해결 상태를 따로 관리할 수 있는지를 먼저 생각해야 한다는 점을 배웠다.

**스택은 단순한 저장소가 아니라 "미해결 관리 도구"** — 아직 답이 없는 원소를 잠시 보관하고, 현재 값이 그 답이 될 수 있으면 해결하는 문제 해결 구조 자체의 핵심 도구라는 점을 느꼈다.

**사고의 전환** — 현재 인덱스의 답을 오른쪽에서 찾는 방식이 아니라, 현재 값이 스택 안의 미해결 인덱스들의 답이 되는지 처리하는 방식으로 사고를 전환해야 했다. 이 발상 전환이 오큰수 문제의 핵심이었다.
