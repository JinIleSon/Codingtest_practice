## 문제 설명

마라톤 참가자 배열 `participant`와 완주자 배열 `completion`이 주어진다.  
완주자는 참가자보다 항상 **1명 적으며**, 완주하지 못한 선수 1명의 이름을 반환해야 한다.

이 문제의 핵심은 **동명이인이 존재할 수 있다**는 점이다.  
단순히 이름이 있는지 없는지만 비교하면 안 되고, **각 이름이 몇 번 등장했는지**를 세어야 한다.

| participant | completion | 정답 |
|-------------|------------|------|
| `["leo", "kiki", "eden"]` | `["eden", "kiki"]` | `"leo"` |
| `["marina", "josipa", "nikola", "vinko", "filipa"]` | `["josipa", "filipa", "marina", "nikola"]` | `"vinko"` |
| `["mislav", "stanko", "mislav", "ana"]` | `["stanko", "ana", "mislav"]` | `"mislav"` |

세 번째 예시처럼 `"mislav"`가 두 번 등장하는 경우가 있기 때문에,  
**문자열 존재 여부가 아니라 등장 횟수의 차이**를 기준으로 판단해야 한다.

---

## 입출력

**입력**
- `participant` : 마라톤 참가자 이름 배열 (1명 이상 100,000명 이하)
- `completion` : 완주자 이름 배열 (참가자보다 1명 적음)

**출력**
- 완주하지 못한 선수의 이름 (문자열)

**제한사항**
- 이름은 1~20자의 알파벳 소문자
- 동명이인이 존재할 수 있음

---

## 헷갈렸던 부분

**1. 이름이 있는지만 비교하면 안 된다**

처음에는 `participant`에 있고 `completion`에 없으면 답이라고 생각하기 쉽다.  
하지만 동명이인이 있으면 이 방식은 바로 깨진다.

```python
participant = ["mislav", "stanko", "mislav", "ana"]
completion  = ["stanko", "ana", "mislav"]
```

`"mislav"`는 `completion`에도 존재하지만, 참가자에는 **2번**, 완주자에는 **1번** 등장한다.  
→ 단순 포함 여부로는 판단 불가.

**2. 이름별 등장 횟수를 세야 한다**

이름별로 `participant`에서 +1, `completion`에서 -1을 하면, 최종적으로 값이 `1`인 이름이 정답이다.

| 이름 | participant | completion | 최종 |
|------|-------------|------------|------|
| mislav | 2 | 1 | **1** |
| stanko | 1 | 1 | 0 |
| ana | 1 | 1 | 0 |

**3. `Counter`를 쓰면 이 과정을 한 줄로 줄일 수 있다**

`collections.Counter`는 리스트의 각 원소 개수를 자동으로 세어주는 자료형이다.

```python
Counter(["a", "b", "a"])
# → Counter({"a": 2, "b": 1})
```

`Counter`끼리 빼면, 개수 차이가 **양수인 항목만** 남는다.

```python
Counter(["a", "a", "b"]) - Counter(["a", "b"])
# → Counter({"a": 1})
```

즉 `participant`에서 `completion`을 빼면 완주하지 못한 선수만 남는다.

## 풀이 접근

### ❌ 처음에 놓치기 쉬운 생각

참가자를 하나씩 돌면서 완주자 배열에 없는 이름을 찾는 방식은 두 가지 문제가 있다.

- 동명이인을 처리할 수 없다
- `in` 연산이나 `remove()`를 반복하면 최악의 경우 `O(n²)`으로 시간 초과

### ✅ 핵심 아이디어 — 빈도 수 비교

이 문제는 이름을 직접 비교하는 문제가 아니라, **이름별 빈도 수를 비교하는 문제**로 바꿔서 보면 된다.

```
1. participant의 이름별 개수를 센다.
2. completion의 이름별 개수를 센다.
3. 두 카운터를 빼서 차이가 남는 이름을 반환한다.
```

```python
Counter(participant) - Counter(completion)
# → 완주하지 못한 선수만 남은 Counter
```

예를 들어:

```python
participant = ["leo", "kiki", "eden"]
completion  = ["eden", "kiki"]

Counter(participant) = {"leo": 1, "kiki": 1, "eden": 1}
Counter(completion)  = {"eden": 1, "kiki": 1}

Counter(participant) - Counter(completion) = {"leo": 1}
```

남은 key 하나를 꺼내 반환하면 끝이다.

---

## 최종 코드

```python
from collections import Counter

def solution(participant, completion):
    # Counter끼리 빼면 value가 0인 항목은 사라지고, 차이가 남은 항목만 남는다
    return list((Counter(participant) - Counter(completion)).keys())[0]
```

**딕셔너리 직접 구현 버전** — Counter 없이 같은 로직

```python
def solution(participant, completion):
    pDict = {}

    for name in participant:
        pDict[name] = pDict.get(name, 0) + 1
    for name in completion:
        pDict[name] = pDict.get(name, 0) - 1

    for key, value in pDict.items():
        if value == 1:
            return key
```

## 복잡도 분석

| 항목 | 복잡도 |
|------|--------|
| 시간 복잡도 | `O(n)` |
| 공간 복잡도 | `O(n)` |

> `n` = 참가자 수 (최대 100,000)

`Counter` 생성, 차집합 계산, key 추출 모두 선형 범위 안에서 처리된다.  
참가자가 최대 100,000명이므로, 해시 기반 카운팅 방식이 적절하다.

## 배운 점

**이 문제는 문자열 비교가 아니라 빈도 수 비교 문제였다**  
이름이 존재하는지 여부만 보면 동명이인 케이스에서 바로 깨진다.  
"무엇을 기준으로 비교해야 하는가"를 먼저 정의하는 습관이 중요하다.

**딕셔너리는 개수 누적 문제에서 강력하다**  
`get(key, 0) + 1` 방식으로 직접 카운팅하면 중복 처리가 자연스럽게 해결된다.  
해시는 검색뿐 아니라 카운팅 문제에도 자주 쓰인다는 점을 다시 확인했다.

**`Counter`는 딕셔너리 카운팅을 더 간결하게 만든다**  
리스트를 바로 개수 형태로 바꿔주고, `Counter - Counter` 연산으로 남는 원소만 바로 구할 수 있다.  
직접 딕셔너리를 구현하는 원리를 이해한 위에서 `Counter`를 쓰는 것이 가장 이상적이다.

**라이브러리를 안다는 것 자체가 실력이다**  
직접 구현 가능한 원리와 표준 라이브러리 활용, 둘 다 갖추는 것이 실전에서 강하다.

---

> 동명이인이 존재한다는 조건 하나 때문에,  
> 이 문제는 단순 탐색이 아니라 **해시 기반 카운팅 문제**로 바뀐다.  
> 핵심은 "존재 여부"가 아니라 **"등장 횟수의 차이"** 를 보는 것이었다.
