## 문제 설명

각 기능의 현재 진도(`progresses`)와 하루 개발 속도(`speeds`)가 주어진다.  
기능은 진도가 100% 이상일 때 배포 가능하며, 앞 기능이 배포되지 않으면 뒤 기능도 함께 기다려야 한다.  
각 배포마다 몇 개의 기능이 배포되는지를 배열로 반환한다.

| progresses | speeds | return |
|------------|--------|--------|
| `[93, 30, 55]` | `[1, 30, 5]` | `[2, 1]` |
| `[95, 90, 99, 99, 80, 99]` | `[1, 1, 1, 1, 1, 1]` | `[1, 3, 2]` |

---

## 입출력

**입력**
- `progresses` : 각 기능의 현재 진도 배열 (길이 100 이하, 값 100 미만)
- `speeds` : 각 기능의 하루 개발 속도 배열 (`progresses`와 길이 동일, 값 100 이하)

**출력**
- 각 배포마다 배포되는 기능의 수를 담은 배열

**제한사항**
- 작업의 개수는 100개 이하
- 배포는 하루에 한 번, 하루의 끝에 이루어짐
- 진도율 95%, 속도 4%/일이면 2일 뒤 배포 (올림 처리)

---

## 헷갈렸던 부분

**1. progresses와 speeds를 함께 pop해야 한다**

기능 하나가 배포될 때 진도와 속도를 동시에 제거해야 한다.  
`progresses`만 pop하면 `speeds`의 인덱스가 어긋나서 이후 zip 연산이 잘못된다.

**2. 앞에서 꺼내야 하므로 deque가 필요하다**

이 문제는 앞에 있는 기능부터 순서대로 배포하는 구조라 front에서 꺼내야 한다.  
일반 리스트의 `pop(0)`은 O(n)이지만, `deque`의 `popleft()`는 O(1)이다.

**3. count가 0일 때는 answer에 넣으면 안 된다**

하루가 지났지만 배포된 기능이 없는 날도 있다.  
`if count`로 0인 경우를 걸러야 한다.

---

## 풀이 접근

### 핵심 아이디어 — 큐로 front를 기준으로 판단

앞에 있는 기능이 100% 이상이어야 배포가 가능한 구조 자체가 FIFO다.  
`progresses[0]`(front)을 기준으로 배포 여부를 판단하므로 큐가 자연스럽게 맞아떨어진다.

```
큐의 핵심 원칙: front만 보고 판단한다.
→ progresses[0] >= 100이면 popleft(), 아니면 다음 날로 넘어간다.
```

**Step 1.** `progresses`와 `speeds`를 `deque`로 변환한다.

**Step 2.** 매일 `zip(progresses, speeds)`로 원소끼리 더해 진도를 업데이트한다.

```python
progresses = deque([x + y for x, y in zip(progresses, speeds)])
```

**Step 3.** `progresses[0] >= 100`인 동안 `popleft()`로 front부터 꺼내며 count를 센다.  
`speeds`도 함께 `popleft()`해야 인덱스가 맞는다.

**Step 4.** count가 1 이상일 때만 `answer`에 추가한다.

---

## 최종 코드

```python
from collections import deque

def solution(progresses, speeds):
    answer = []
    # 앞에서부터 볼 거기 때문에 deque로 사용
    progresses = deque(progresses)
    speeds = deque(speeds)

    while len(progresses) > 0:
        count = 0
        # 하루치 진도 업데이트: 같은 인덱스끼리 더함
        progresses = deque([x + y for x, y in zip(progresses, speeds)])
        # front부터 100 이상인 기능 꺼내기
        while len(progresses) > 0 and progresses[0] >= 100:
            progresses.popleft()
            speeds.popleft()  # progresses와 speeds 함께 pop해야 함
            count += 1
        # count가 있을 때만 개수 넣기
        if count:
            answer.append(count)

    return answer
```

## 복잡도 분석

| 항목 | 복잡도 |
|------|--------|
| 시간 복잡도 | `O(n * d)` |
| 공간 복잡도 | `O(n)` |

> `n` = 기능 수 (최대 100), `d` = 최대 소요 일수

매일 zip으로 전체를 순회하므로 일수만큼 반복된다.  
`popleft()`는 O(1)이라 꺼내는 비용은 무시 가능하다.  
n이 100 이하로 작아 실질적으로 빠르게 동작한다.

## 배운 점

**이 문제가 큐인 이유**  
앞 기능이 완료되어야 뒤 기능도 배포 가능한 구조가 FIFO와 일치한다.  
front를 기준으로 판단하고 front에서 꺼내므로 큐가 자연스럽게 맞아떨어진다.

**`deque`를 쓰는 이유**  
리스트의 `pop(0)`은 나머지를 전부 당겨야 해서 O(n)이다.  
`deque`는 양쪽에 포인터를 가진 구조라 `popleft()`가 O(1)이다.  
front에서 자주 꺼내야 하는 큐 구현에는 `deque`가 적합하다.

**`zip`으로 두 배열을 동시에 순회**  
`zip(progresses, speeds)`는 같은 인덱스의 원소를 튜플로 묶어준다.  
인덱스 없이 `x + y`로 바로 접근할 수 있어 코드가 간결해진다.

**`progresses`와 `speeds`를 반드시 함께 pop해야 한다**  
하나만 pop하면 이후 zip에서 인덱스가 어긋난다.  
두 배열이 항상 같은 인덱스를 가리켜야 하므로 동시에 관리해야 한다.

**`if count`로 0 필터링**  
배포가 없는 날도 while을 돌기 때문에 count가 0인 경우가 생긴다.  
파이썬에서 0은 falsy라 `if count`로 간단하게 걸러낼 수 있다.

---

> 이 문제의 핵심은 **앞에서부터 순서대로 처리**해야 한다는 구조다.  
> front를 기준으로 판단하고 front에서 꺼내는 패턴이 큐와 정확히 맞아떨어진다.
