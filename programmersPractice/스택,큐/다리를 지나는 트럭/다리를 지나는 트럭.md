## 문제 설명

트럭들이 일차선 다리를 정해진 순서대로 건넌다.  
다리에는 최대 `bridge_length`대까지 올라갈 수 있고, 무게 합이 `weight`를 초과하면 진입 불가다.  
모든 트럭이 다리를 건너는 데 걸리는 최소 시간을 반환한다.

| bridge_length | weight | truck_weights | return |
|---------------|--------|---------------|--------|
| `2` | `10` | `[7, 4, 5, 6]` | `8` |
| `100` | `100` | `[10]` | `101` |
| `100` | `100` | `[10, 10, 10, 10, 10, 10, 10, 10, 10, 10]` | `110` |

---

## 입출력

**입력**
- `bridge_length` : 다리에 올라갈 수 있는 최대 트럭 수 (1 이상 10,000 이하)
- `weight` : 다리가 견딜 수 있는 최대 무게 (1 이상 10,000 이하)
- `truck_weights` : 트럭별 무게 배열 (길이 1 이상 10,000 이하, 각 무게는 1 이상 weight 이하)

**출력**
- 모든 트럭이 다리를 건너는 데 걸리는 최소 시간 (초)

**제한사항**
- 매 초마다 트럭은 한 칸씩 이동한다
- 대기 트럭도 1초에 한 대씩만 진입할 수 있다
- 다리에 완전히 오르지 않은 트럭의 무게는 무시한다

---

## 헷갈렸던 부분

**1. 다리를 자료구조로 어떻게 표현할지**

다리를 `bridge_length` 크기의 큐로 직접 모델링했다.  
초기값은 전부 `0`(빈 칸)으로 채우고, 매 초마다 앞에서 하나 빼고 뒤에서 하나 넣는 방식이다.  
트럭이 없는 칸은 `0`으로 채워 무게 합 계산에서 자연스럽게 제외된다.

**2. 무게 합을 매 루프마다 `sum()`으로 계산하면 안 된다**

`sum(bridge_queue)`는 deque 전체를 순회하므로 O(n)이다.  
루프마다 호출하면 전체가 O(n²)이 되어 데이터가 클수록 시간 초과가 발생한다.  
별도 변수 `current_weight`를 두고 빼고 더하는 방식으로 O(1)을 유지해야 한다.

**3. `popleft()` 직후 무게 변수를 즉시 반영해야 한다**

`popleft()`로 값을 꺼낸 뒤 `current_weight`를 바로 갱신하지 않으면,  
if문에서 이미 빠진 값이 포함된 채로 무게 비교가 이루어져 답이 틀린다.  
꺼낸 즉시 `current_weight -= removed`를 처리해야 한다.

## 풀이 접근

### 핵심 아이디어 — 다리를 큐로 모델링

트럭이 정해진 순서대로 진입하고 앞에서부터 빠져나오는 구조 자체가 FIFO다.  
다리를 `bridge_length` 크기의 큐로 직접 표현하면 매 초의 상태를 자연스럽게 시뮬레이션할 수 있다.

```
큐의 핵심 원칙: 매 초마다 앞에서 하나 빼고, 뒤에서 하나 넣는다.
→ 무게 조건을 만족하면 트럭을, 아니면 0을 넣는다.
```

**Step 1.** `truck_weights`를 `deque`로 변환하고, `bridge_queue`를 `0`으로 `bridge_length`만큼 채운다.

**Step 2.** 매 루프마다 `popleft()`로 앞에서 하나 꺼내고 `current_weight`에서 즉시 뺀다.

**Step 3.** 다음 트럭이 진입 가능한지(`current_weight + truck_weights[0] <= weight`) 확인한다.  
가능하면 트럭을 넣고 `current_weight`에 더한다. 불가능하면 `0`을 넣는다.

**Step 4.** `current_weight > 0` 이거나 `truck_weights`가 남아있으면 반복한다.

## 최종 코드

```python
from collections import deque

def solution(bridge_length, weight, truck_weights):

    truck_weights = deque(truck_weights)

    bridge_length_queue = deque()
    for i in range(bridge_length):
        bridge_length_queue.append(0)

    time = 0
    current_weight = 0

    # bridge_length_queue에 값이 들어가 있을 때도 돌아가게끔 해야 함(다리에만 남아있을 때도 가정)
    # truck_weights가 비면 False, 아니면 True
    while current_weight or truck_weights:
        # 다리에서 빠지는 무게 current_weight에서 빼기
        current_weight -= bridge_length_queue[0]
        bridge_length_queue.popleft()
        # truck_weights가 있을 때만 if문에 들어가게끔 함(오른쪽 인덱스 참조 때문에)
        # 현재 문제: sum(bridge_length_queue)를 두 번, 매 루프마다 호출하면 무거워진다. 모든 수가 루프 한 번에 전체 호출됨
        # 개별의 변수(current_weight)로 전체 무게를 추적하면 됨
        if truck_weights and weight >= current_weight + truck_weights[0]:
            current_weight += truck_weights[0]
            bridge_length_queue.append(truck_weights.popleft())
        else:
            bridge_length_queue.append(0)
        time += 1

    return time
```

## 복잡도 분석

| 항목 | 복잡도 |
|------|--------|
| 시간 복잡도 | `O(n + t)` |
| 공간 복잡도 | `O(bridge_length)` |

> `n` = 트럭 수, `t` = 총 소요 시간

`popleft()`와 `append()`는 모두 O(1)이다.  
`current_weight`를 변수로 관리해 `sum()` 호출을 제거했으므로 루프당 비용은 O(1)이다.  
bridge_queue의 크기는 항상 `bridge_length`로 고정된다.

## 배운 점

**이 문제가 큐인 이유**  
트럭이 정해진 순서대로 진입하고 먼저 들어온 트럭이 먼저 나가는 구조가 FIFO와 일치한다.  
다리 자체를 큐로 표현하면 매 초의 상태를 직관적으로 시뮬레이션할 수 있다.

**다리를 `0`으로 채운 큐로 모델링하는 이유**  
다리의 각 칸을 명시적으로 표현하면 빈 칸도 자연스럽게 무게 계산에 포함된다.  
트럭이 없는 자리는 `0`이므로 별도 처리 없이 `current_weight`에 영향을 주지 않는다.

**`sum()` 대신 변수로 무게를 관리해야 하는 이유**  
`sum(deque)`는 O(n)이라 루프마다 호출하면 O(n²)이 된다.  
변화량만 반영하는 방식(`current_weight += / -=`)은 O(1)이라 데이터가 커도 빠르게 동작한다.  
슬라이딩 윈도우 문제에서도 자주 쓰이는 패턴이다.

**`popleft()` 직후 즉시 반영해야 하는 이유**  
꺼낸 값을 바로 `current_weight`에서 빼지 않으면, if문에서 이미 빠진 값이 포함된 채로 비교가 이루어진다.  
상태 변화는 발생한 즉시 반영해야 이후 로직이 정확하게 동작한다.

---

> 이 문제의 핵심은 **다리를 고정 크기의 큐로 직접 모델링**하는 발상이다.  
> 매 초 앞에서 빼고 뒤에서 넣는 단순한 구조 안에서, 무게 조건만 체크하면 시뮬레이션이 완성된다.
